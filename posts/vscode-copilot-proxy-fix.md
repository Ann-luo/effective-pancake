# VS Code GitHub Copilot 代理连接错误：排查与修复完全指南

## 一、你遇到了什么问题？

当你在 VS Code 中使用 GitHub Copilot 时，可能会看到这样的错误：

```
api.github.com/copilot_internal/user - error GET Failed to fetch
net::ERR_PROXY_CONNECTION_FAILED
```

翻译成人话就是：**VS Code 试图通过一个"中介"（代理服务器）去访问 GitHub，但这个中介已经死掉了，所以请求永远到不了目的地。**

就像你想去餐厅吃饭，结果导航把你导到一条封掉的路，你就卡在那儿了。

---

## 二、为什么会这样？

VS Code 在启动时，会从多个地方"继承"代理设置。常见的代理来源有：

| 来源 | 说明 |
|------|------|
| Windows 系统代理设置 | 设置 → 网络和互联网 → 代理 |
| 系统环境变量 | `HTTP_PROXY` / `HTTPS_PROXY` |
| VS Code 自己的设置 | `settings.json` 中的 `http.proxy` |
| Copilot 扩展的独立设置 | GitHub Copilot 扩展的代理配置 |

**最坑的地方是**：你可能从来没有手动设置过代理，但某些软件（如科学上网工具、公司 VPN）会在你不注意的时候偷偷开启系统代理。当这些工具关掉后，代理开关却还开着，导致 VS Code 每次启动都去连一个不存在的中介，然后报错。

## 三、逐步排查指南（从最重要到最冷门）

> ⚠️ 每做完一步，都**完全退出 VS Code 再重新打开**，才能验证是否生效。

### 步骤 1：检查 Windows 系统代理（解决 80% 的问题）

这是最常见的原因，务必优先检查。

1. 打开 Windows **设置**（按 `Win + I`）
2. 点击 **网络和互联网** → **代理**
3. 确保以下三项**全部关闭**，且没有任何地址填写：
   - ☐ 自动检测设置 → **关**
   - ☐ 使用设置脚本 → **关**
   - ☐ 手动设置代理 → **关**（地址和端口都要为空）

![示意图：三项全部关闭]

改完后，重启 VS Code 测试。

---

### 步骤 2：清理系统环境变量

有些程序会在系统层面写入代理环境变量，需要手动删掉。

1. 在 Windows 搜索框输入 **"编辑系统环境变量"** 并打开
2. 点击右下角的 **环境变量** 按钮
3. 在 **用户变量** 和 **系统变量** 两个列表中，分别查找：
   - `HTTP_PROXY`
   - `HTTPS_PROXY`
   - `ALL_PROXY`
   - `NO_PROXY`
4. 如果找到这些变量，**选中 → 删除 → 确定**

删完后重启 VS Code。

---

### 步骤 3：在 VS Code 内置终端确认环境变量

有时候系统变量删了，VS Code 内部还残留着，需要进去看一眼。

1. 在 VS Code 中按 **`Ctrl + ``** 打开内置终端
2. 输入以下命令并回车：

```powershell
$env:HTTP_PROXY
$env:HTTPS_PROXY
```

如果输出是**空白**，说明没问题。如果还有输出，说明没删干净，回到步骤 2 再检查。

---

### 步骤 4：用命令行参数强制禁用代理（临时方案 / 定位利器）

如果以上步骤都无效，或者你想快速确认问题到底是不是代理导致的，用这个方法：

1. **完全关闭所有 VS Code 窗口**
2. 打开 PowerShell 或命令提示符
3. 输入以下命令并回车：

```powershell
code --proxy-server=""
```

这个命令的意思是：**"启动 VS Code，并且强制它不用任何代理，直接联网。"**

如果这样启动后 Copilot 恢复正常，那就 100% 确认是代理的问题了。

> ⚠️ 这个命令**只对当次启动有效**，关了 VS Code 下次直接打开还会恢复原状。

---

### 步骤 5：检查 Copilot 扩展自身的代理设置

1. 在 VS Code 中，点击左下角齿轮图标 ⚙️ → **设置**
2. 搜索框输入 `Copilot proxy`
3. 找到 **GitHub Copilot: Proxy** 这一项
4. 确保它是**空白的**，或者选择 **"Use VS Code proxy settings"**（使用 VS Code 的代理设置）

不要在这里手动填任何代理地址。

---

### 步骤 6：兜底手段

如果以上全部无效，可能是公司网络安装了强制代理软件或根证书，劫持了所有 HTTPS 流量。这时可以：

- 联系公司网络管理员
- 或者在 VS Code 的 `http.proxy` 中填写一个**可用的**代理地址（比如你本机的科学上网工具）

## 四、如何永久解决？（两种方案）

### 方案 A：根治 —— 清除代理源头（推荐 ✅）

完成上面步骤 1 和步骤 2 —— 关掉 Windows 系统代理开关、删掉系统环境变量中的代理变量，问题就一劳永逸了。

以后直接双击 VS Code 图标打开就能正常使用 Copilot，不需要任何额外操作。

---

### 方案 B：绕过 —— 修改快捷方式自带参数

如果你暂时找不到代理来源（或者不想动它），可以修改 VS Code 的快捷方式，让它每次启动都自动带上禁用代理的参数：

1. 在桌面找到 VS Code 快捷方式，**右键 → 属性**
2. 在 **目标** 一栏的路径最后，加上一个空格，然后粘贴：

```
--proxy-server=""
```

例如修改后变成：

```
"C:\Users\你的用户名\AppData\Local\Programs\Microsoft VS Code\Code.exe" --proxy-server=""
```

3. 点 **确定** 保存
4. 如果任务栏也有 VS Code 图标，先取消固定，然后从桌面改好的快捷方式重新拖到任务栏

这样以后每次从这个快捷方式启动 VS Code，都会自动禁用代理。

## 六、常见问答 FAQ

### Q1：为什么 curl 能通，VS Code 却不行？

因为 `curl` 是在命令行里运行的，它不一定遵循 Windows 的图形界面代理设置。而 VS Code 是基于 Chromium 内核（Electron）构建的，它会严格遵循 Windows 系统的代理设置。两者走的"路"不一样。

### Q2：我从来没设置过代理，为什么会这样？

很多软件（如某些 VPN、抓包工具、加速器）会在安装或运行时悄悄打开系统代理开关。当你卸载这些软件或它们异常退出后，开关却还开着，就产生了"幽灵代理"。

### Q3：这个修复对其他软件有效吗？

如果其他软件（如 Edge 浏览器、Chrome 浏览器）也报类似的代理错误，关闭 Windows 系统代理同样能修复它们。
