## 前言

今天让 AI 帮我装了个叫「mattpocock/skills」的东西，然后搭了个猜数字小游戏。整个过程 AI 一边干活一边解释，我从一脸懵逼到勉强理解。这篇日记把全过程记下来。

**阅读前提**：你知道电脑能开机，知道浏览器能上网，其他啥也不懂。够用了。

## 第一章：Skills 是什么

### 我第一次听说这个词

AI 跟我说：「我帮你装 skills」。

我：skill 不是"技能"的意思吗？

AI 解释：**Skills 就是给 AI 装的技能包**。装了 skill 之后，AI 干活更有章法。

打个比方：
- 没装 skill 的 AI：你跟它说「修个 bug」，它可能直接上手改，改完就忘
- 装了 skill 的 AI：先复现 bug → 缩小范围 → 推测原因 → 精准修复 → 验证，每一步都有记录

### 24 个技能包分别是干嘛的

mattpocock/skills 仓库里有 24 个 skill，分五大类：

**写代码用的（engineering）：**

| Skill | 干嘛的 |
|-------|--------|
| diagnose | 专业 debug 流程：复现→定位→修复→验证 |
| tdd | 先写测试再写代码 |
| to-issues | 把大活拆成小任务 |
| to-prd | 把想法整理成需求文档 |
| triage | 给任务排优先级 |
| review | 审查代码 |
| prototype | 快速搭原型 |
| setup-pre-commit | 提交前自动检查错误 |
| git-guardrails | 防止危险 git 操作 |
| zoom-out | 跳出细节看全局 |

**写作用的：**

| Skill | 干嘛的 |
|-------|--------|
| writing-beats | 像游戏分支一样写文章 |
| writing-fragments | 收集零散想法 |
| writing-shape | 打磨成可发布的文章 |
| edit-article | 编辑改进文章 |

**思维/沟通用的：**

| Skill | 干嘛的 |
|-------|--------|
| grill-me | 像面试官一样追问你的计划 |
| caveman | 极简模式，省字数 |
| teach | 帮你学新技术 |
| handoff | 打包对话交给另一个 AI |
| obsidian-vault | 管理 Obsidian 笔记 |

**今天的教训**：不用一次全记住。知道这东西大概干嘛的就行，用的时候再说。

## 第二章：初始化仓库——到底在初始化什么

### 什么是"仓库"

仓库就是你的项目文件夹。桌面上那个 `codex-demo` 文件夹，里面有代码、文档、配置，一整个叫仓库。

### 初始化干了什么

我让 AI 跑 `setup-matt-pocock-skills`，它在项目里创建了几个文件：

```
codex-demo/
├── AGENTS.md               ← 给 AI 看的项目说明书
└── docs/agents/
    ├── issue-tracker.md    ← 规定任务存哪
    ├── triage-labels.md    ← 任务标签对照表
    └── domain.md           ← 告诉 AI 怎么看文档
```

**大白话**：就像新公司第一天报到——HR 告诉你工位在哪、WiFi 密码、报销找谁。初始化就是让 AI「报到」，知道这个项目的规矩。

### 初始化问了我三个问题

| 问题 | 我选的 | 意思 |
|------|--------|------|
| 任务存哪？ | 本地 | 任务存在项目文件夹里，不依赖外部网站 |
| 标签叫啥？ | 默认 | 五个状态：待评估/等补充/可自动做/需人做/不做 |
| 文档结构？ | 单上下文 | 一份说明书管整个项目 |

以后在这个项目里用任何一个 skill，AI 都会先看这三个配置。

## 第三章：做猜数字小游戏的完整流程

### 第一步：to-prd（写需求文档）

AI 让我说想做什么，我说「做个小游戏」。

AI 就把这个模糊想法写成了正式需求文档，存到 `.scratch/number-guessing-game/PRD.md`。

文档里写了：
- **问题**：现在的项目没有能玩的东西
- **方案**：加猜数字游戏，1-100 随机数，玩家猜
- **7 个用户故事**：比如"输入数字提交猜测""看到高低反馈"
- **不做的东西**：排行榜、多人模式、手机适配

PRD 文件第一行是 `Status: ready-for-agent`——意思是 AI 评估后认为可以自己干。

### 第二步：to-issues（拆任务）

把需求拆成 3 个小任务：

| # | 任务 | 说明 |
|---|------|------|
| 01 | 游戏逻辑 | 写 `checkGuess(猜的数, 目标数)`，返回高了/低了/对了 |
| 02 | API 接口 | 服务器上开接口，接收猜测、返回结果 |
| 03 | 前端页面 | HTML 页面，输入框、按钮、显示结果 |

每个任务存在 `.scratch/number-guessing-game/issues/` 下。

### 第三步：一个接一个实现

**任务 1：** 写了 `src/game.ts`——就两个函数：
- `checkGuess(guess, target)` → 返回高了/低了/对了
- `createGame()` → 生成随机数，重开游戏

**任务 2：** 改了 `src/index.ts`，加了两个网络接口。网页点"猜"，浏览器发请求给服务器，服务器算完返回结果。

**任务 3：** 写了 `public/index.html`——深色背景、输入框、Guess 按钮、New Game 按钮。

### 最终项目长这样

```
codex-demo/
├── AGENTS.md                       ← AI 的项目说明书
├── .scratch/                       ← 任务追踪
│   └── number-guessing-game/
│       ├── PRD.md                  ← 需求文档
│       └── issues/
│           ├── 01-game-logic.md
│           ├── 02-api-endpoints.md
│           └── 03-frontend-ui.md
├── docs/agents/                    ← 初始化配置
├── src/
│   ├── index.ts                    ← 服务器
│   ├── game.ts                     ← 游戏逻辑
│   └── game.test.ts               ← 测试
└── public/
    └── index.html                  ← 游戏页面
```

## 第四章：服务器、前端、后端——这三个词困扰了我一整天

### 服务器是什么

我问 AI：「为什么猜数字还要服务器？」

AI 打比方：

> 你去餐厅。你是客人，厨房是服务器。你说"来碗面"（发请求），厨房做好端给你（返回结果）。服务器就是那台等着你发请求、给你返回内容的电脑。

我追问：「那我电脑就是服务器？localhost 是啥？」

AI：**`localhost` 就是这台电脑自己。** `localhost:3000` = 我自己的电脑，门牌号 3000。

等于在自己家开了个餐厅——自己点菜，自己炒，自己吃。一台电脑既是客人也是厨房。

### 前端和后端

猜数字游戏分了两坨：

```
前端（public/index.html）         后端（src/index.ts）
─────────────────────────         ─────────────────────
  👁 你看到的页面                  🤖 你看不到的服务器
  输入框、按钮、颜色               随机生成目标数字
  点按钮发请求给后端               判断猜高了还是低了
  显示结果                         记着你猜了几次
```

- **前端** = 能看见能摸的（界面、按钮、颜色）
- **后端** = 看不见但在干活的（计算、判断、存储）

### 为什么双击版不需要服务器

AI 后来做了个 `guess-the-number.html` 放桌面，双击就能玩。

原理：**把后端干的活全塞进前端**——前后端合并到一个文件。

```
服务器版：                      双击版：
浏览器 ←→ 服务器                 浏览器自给自足
（前后端分离）                   （前后端合并）
```

### 服务器怎么关

- 找到黑窗口（命令行），按 `Ctrl + C`
- 或者让 AI 关：它跑一条命令就停了

## 第五章：这堆 skills 到底改变了什么

### 有 skill vs 没 skill

| | 没 skill | 有 skill |
|---|----------|----------|
| 说"做个游戏" | 直接撸代码 | 先写需求→拆任务→再动手 |
| 任务管理 | 没有 | 存 `.scratch/` 里，随时看 |
| 做法 | 可能跳步骤 | 按固定流程来 |

### 今天实际走过的完整流程

```
我的想法："做个小游戏"
    ↓
初始化项目配置（setup-matt-pocock-skills，只做一次）
    ↓
写成需求文档（to-prd）
    ↓
拆成小任务（to-issues）
    ↓
一个个实现（写代码）
    ↓
玩！
```

以后新项目，流程一模一样。初始化只做一次。

## 第六章：我的吐槽 & 真实感受

### 开头真的看不懂

- AI 说"skill""repo""PRD""issue tracker""triage label"——我：？？？
- 全靠 AI 打比方（餐厅厨房、公司报到、淘宝订单状态）才勉强理解
- 后来发现：**不用一次全懂。知道大概干嘛的就行，用的时候再说**

### 实际没那么可怕

掰开了就是：
- **Skill** = AI 的做事规矩
- **仓库** = 项目文件夹
- **初始化** = 第一次用前配好设置
- **PRD** = 把想法写成正式文档
- **Issue** = 拆分出的小任务
- **前端** = 能看见的
- **后端** = 看不见但在干活的
- **服务器** = 等着你发请求的电脑

### 最重要的

AI 不是一次性工具。装了 skills 之后，每一步都有记录。打开项目文件夹 `.scratch/number-guessing-game/`，能看到整个游戏的来龙去脉——需求→任务→代码，清清楚楚。
